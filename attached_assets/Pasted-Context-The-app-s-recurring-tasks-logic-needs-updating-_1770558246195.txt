Context: The app’s recurring tasks logic needs updating to work with the unified tasks structure. Previously, recurrence may have been handled separately for personal reminders or project tasks. Prompt Instructions:

Remove Type-specific Conditions: Go through the recurring tasks implementation (both backend scheduling logic and any frontend UI for recurrence). Eliminate any branches that check for different task types. For example, if code was like if (isPersonalReminder(task)) { ... } else if (isProjectTask(task)) { ... }, remove these distinctions. All tasks are now just Task with a repeat pattern, so the recurrence logic should treat them uniformly.

Use Unified repeat Field: Ensure the recurrence scheduler (cron job or background worker that generates recurring tasks) looks at the unified tasks table. It should query for tasks that have a repeat value not equal to "none" (or however a non-recurring task is marked). This query no longer needs to union multiple tables – it’s a single query on tasks (e.g., SELECT * FROM tasks WHERE repeat IS NOT NULL AND repeat != 'none' AND deletedAt IS NULL). The result will include all tasks that need recurrence handling, regardless of their origin.

Consistency in Recurrence Handling: Implement the recurrence according to the repeat pattern for each task:

If task.repeat = 'daily', when the task is completed (or at midnight if using date-based generation), create the next occurrence of the task due the next day.

If repeat = 'weekly', generate the next task one week from the previous due date or completion date.

Similarly handle 'monthly' or other supported intervals. The key is that the logic uses the task’s own data to decide when and how to recur, not any external table.

Each new occurrence should be inserted into the tasks table as a fresh row, copying relevant fields (title, description, sourceType, sourceId, etc.) and adjusting dates. By inserting a new row for each occurrence, we preserve the history of completed tasks and create a distinct task for each recurrence. This also means id will be new for the new task, but it will carry over the same sourceType/sourceId (so, for example, a recurring personal task remains personal, a recurring project task stays associated with its project).

For instance, if personal reminders had a boolean for “isRecurring” and an interval, translate that into the repeat enum (e.g., set repeat = 'weekly' if it was marked weekly). Ensure no recurrence info is lost.

This approach ensures at most one outstanding instance of a recurring task at a time, and the task list remains tidy.
